# 微型环境数据记录与 Shell 交互终端

本工程面向 STM32F103C8，使用 STM32F10x 标准外设库和 Keil ARMCC 5。固件已经按需求实现非阻塞前台调度、后台采集、异常日志、串口 Shell、看门狗和空闲睡眠。

## 硬件连接

| 功能 | 外设 | 引脚 |
| --- | --- | --- |
| 光敏模拟量 | ADC1 通道 0 + DMA1 通道 1 | PA0 |
| W25Q64 片选/时钟/MISO/MOSI | SPI1 | PA4/PA5/PA6/PA7 |
| 串口 TX/RX | USART1，9600-8-N-1 | PA9/PA10 |
| MPU6050 SCL/SDA | I2C2，100 kHz | PB10/PB11 |
| OLED SCL/SDA | 软件 I2C | PB8/PB9 |
| 报警 LED | GPIO，低电平点亮 | PA1 |

OLED 驱动使用地址字节 `0x78`，对应常见的 SSD1306 7 位地址 `0x3C`。W25Q64 检测的 JEDEC ID 为 `EF4017`。

## 运行逻辑

- SysTick 提供 1 ms 系统时间，不在主循环中调用阻塞式延时。
- 每 50 ms 检查一次以 `\r\n` 结束的串口命令。
- 每 100 ms 读取 MPU6050 并检查倾斜和冲击。
- 每 500 ms 刷新 OLED 时间、ADC、状态和日志空间。
- 主循环末尾喂独立看门狗，看门狗超时约 4 秒。
- 连续 5 分钟无串口活动且没有报警时进入 `__WFI()` 睡眠；任意串口字符都会通过 RXNE 中断唤醒系统。

当前光敏电阻分压电路在光线增强时 ADC 数值下降，因此显示百分比采用反向换算：ADC 越低，亮度百分比越高。

MPU6050 加速度量程配置为正负 16 g。默认假设设备静止时 Z 轴朝上：X 或 Y 轴绝对值达到约 0.707 g 判定为严重倾斜，任一轴达到 2.5 g 判定为冲击。报警恢复需要连续 2 秒处于正常状态，避免同一次事件反复写入日志。

## 串口 Shell

串口参数为 9600 baud、8 数据位、无校验、1 停止位、无硬件流控。固件同时接受回车、换行或 CRLF；常见串口助手中输入命令后直接点击“发送”即可。

```text
sys_info
get_rtc
set_rtc -t 16:30:00
read_log -n 10
format_flash
help
```

- `sys_info`：显示运行毫秒数、W25Q64 ID、日志数量、剩余空间和看门狗复位状态。
- `get_rtc`：显示当前完整日期时间。
- `set_rtc -t hh:mm:ss`：保留当前日期，只更新时间。
- `read_log -n N`：按时间顺序输出最近 N 条日志。
- `format_flash`：擦除日志扇区，删除全部日志。

接收缓冲区最多保存 95 个命令字符，超长命令会返回错误，不会越界写入。

## 日志格式与策略

日志使用 W25Q64 起始地址 `0x000000` 的一个独立 4 KB 扇区。每条二进制记录为 32 字节，包含：

- 魔数和递增序号；
- RTC 时间戳；
- ADC 原始值；
- 三轴加速度原始值；
- 倾斜/冲击类型；
- CRC16 校验。

扇区容量为 128 条。为保留黑匣子证据，写满后不会自动覆盖旧记录；系统会报告日志写入失败，用户确认后可执行 `format_flash`。

## OLED 页面

```text
TIME 16:30:00
LIGHT 2048 050%
STATE ACTIVE
LOG 003 F4000
```

状态可能为 `ACTIVE`、`WARNING`、`SLEEP` 或 `MPU ERROR`。

## 构建与硬件验证

打开 `Project.uvprojx` 后构建 Target 1。工程启用了 HEX 输出，生成文件位于 `Objects` 目录。

提交前已使用 ARMCC 5.06 update 5 全量重编译：0 Error、0 Warning。由于当前环境无法连接实体开发板，烧录后仍需依次确认：

1. `sys_info` 返回 W25Q64 ID `EF4017`，OLED 不显示硬件错误；
2. PA0 输入变化时 OLED 的 ADC 值连续更新；
3. 倾斜或快速移动开发板后串口出现 `[WARN]` 且 `read_log` 能读回记录；
4. `set_rtc`、断电后的 VBAT 保持及 RTC 走时误差符合实际板卡；
5. 5 分钟空闲睡眠、UART 唤醒以及人为制造死循环后的 IWDG 复位行为正常。
